18//! Bindings for the Kubernetes client API, generated from the OpenAPI spec.
19//!
20//! Each supported version of Kubernetes is represented by a feature name (like `v1_9`). Only one such feature can be enabled at a time.
21//!

34//! # Examples
35//!
36//! ## Resources
37//!
38//! This example creates an instance of [`api::core::v1::PodSpec`] with no other properties set, and pretty-prints it.
39//!
40//! ```rust
41//! use k8s_openapi::api::core::v1 as api;
42//!
43//! fn main() {
44//!     let pod_spec: api::PodSpec = Default::default();
45//!     println!("{pod_spec:#?}");
46//! }
47//! ```
48//!
49//!
50//! # Crate features
51//!
52//! This crate contains several `v1_*` features. Enabling one of the `v1_*` features selects which version of the Kubernetes API server this crate should target.
53//! For example, enabling the `v1_23` feature means the crate will only contain the API exposed by Kubernetes 1.23. It will not expose API
54//! that were removed in 1.23 or earlier, nor any API added in 1.24 or later.
55//!
56//! One and only one of the `v1_*` features must be enabled at the same time, otherwise the crate will not compile. This ensures that all crates in the crate graph
57//! use the same types. If it was possible for one library crate to use `api::core::v1::Pod` corresponding to v1.50 and another to use the type
58//! corresponding to v1.51, an application would not be able to use the same `Pod` value with both.
59//!
60//! Thus, it is recommended that only application crates must enable one of the `v1_*` features, corresponding to the version of Kubernetes
61//! that the application wants to support.
62//!
63//! ```toml
64//! # For application crates
65//!
66//! [dependencies]
67//! k8s-openapi = { version = "...", features = ["v1_50"] }
68//! ```
69//!
70//! If you're writing a library crate, your crate *must not* enable any features of `k8s-openapi` directly. The choice of which feature to enable
71//! must be left to any application crates that use your library. This ensures that all `k8s-openapi`-using dependencies in that application crate's dependency graph
72//! use the same set of `k8s-openapi` types and are interoperable.
73//!
74//! If your library crate has tests or examples, you should also add a dev-dependency on `k8s-openapi` in addition to the direct dependency,
75//! and enable a version feature only for that dev-dependency.
76//!
77//! ```toml
78//! # For library crates
79//!
80//! [dependencies]
81//! k8s-openapi = "..."
82//!
83//! [dev-dependencies]
84//! k8s-openapi = { version = "...", features = ["v1_50"] }
85//! ```
86//!
87//! However, commands like `cargo check` and `cargo doc` do not build dev dependencies, so they will not enable the feature and will fail to build. There are two ways
88//! you can resolve this:
89//!
90//! 1. Add a feature to your library that enables one of the k8s-openapi `v1_*` features, and then remember to enable this feature when running such commands.
91//!
92//!    ```toml
93//!    [features]
94//!    __check = ["k8s-openapi/v1_50"]
95//!    ```
96//!
97//!    ```sh
98//!    $ cargo check --features __check
99//!    ```
100//!
101//! 1. Define the `K8S_OPENAPI_ENABLED_VERSION` env var when running such commands:
102//!
103//!    ```sh
104//!    $ K8S_OPENAPI_ENABLED_VERSION=1.50 cargo check
105//!    ```
106//!
107//!
108//! # Conditional compilation
109//!
110//! As the previous section explained, library crates must not enable any version features in their `k8s-openapi` dependency. However, your library crate may
111//! need to know about which version gets selected eventually.
112//!
113//! For example:
114//!
115//! 1. Your crate creates a `PodSpec` and wants to set the `host_users` field. This field is only available in Kubernetes 1.25+,
116//!    so you want your crate to fail to compile if a lower feature was enabled.
117//!
118//! 1. Your crate creates a `PodSpec` and wants to set the `host_users` field, but it's okay to not set it when compiling for older versions.
119//!
120//! There are two ways for your crate to determine which feature of `k8s-openapi` is enabled:
121//!
122//! 1. The `k8s-openapi` crate exports [`k8s_if_*` macros,](#macros) which either expand to their contents or don't. See the docs of the macros for more details.
123//!
124//!    With these macros, the two cases above would be solved like this:
125//!
126//!    - ```rust,ignore
127//!      // The compile_error!() is only emitted if 1.24 or lower is selected.
128//!      k8s_openapi::k8s_if_le_1_24! {
129//!          compile_error!("This crate requires the v1_25 (or higher) feature to be enabled on the k8s-openapi crate.");
130//!      }
131//!
132//!      ...
133//!
134//!      let pod_spec = k8s_openapi::api::core::v1::PodSpec {
135//!          host_users: ...,
136//!          ...
137//!      };
138//!      ```
139//!
140//!    - ```rust,ignore
141//!      let mut pod_spec = k8s_openapi::api::core::v1::PodSpec {
142//!          ...
143//!      };
144//!
145//!      k8s_openapi::k8s_if_ge_1_25! {
146//!          pod_spec.host_users = ...;
147//!      }
148//!      ```
149//!
150//! 1. The `k8s-openapi` crate emits the selected version number as metadata that your crate can read in a build script
151//!    from the `DEP_K8S_OPENAPI_*_VERSION` env var.
152//!
153//!    ```rust,no_run
154//!    // Your crate's build.rs
155//!
156//!    fn main() {
157//!        let k8s_openapi_version: u32 =
158//!            std::env::vars_os()
159//!            .find_map(|(key, value)| {
160//!                let key = key.into_string().ok()?;
161//!                if key.starts_with("DEP_K8S_OPENAPI_") && key.ends_with("_VERSION") {
162//!                    let value = value.into_string().ok()?;
163//!                    Some(value)
164//!                }
165//!                else {
166//!                    None
167//!                }
168//!            }).expect("DEP_K8S_OPENAPI_*_VERSION must have been set by k8s-openapi")
169//!            .parse().expect("DEP_K8S_OPENAPI_*_VERSION is malformed");
170//!
171//!        // k8s_openapi_version has the format 0x00_MM_NN_00.
172//!        //
173//!        // - MM is the major version.
174//!        // - NN is the minor version.
175//!        //
176//!        // Thus, if the v1_25 feature was enabled, k8s_openapi_version would be 0x00_01_19_00
177//!
178//!        // The build script can now do arbitrary things with the information.
179//!        // For example, it could define custom cfgs:
180//!        if k8s_openapi_version >= 0x00_01_19_00 {
181//!            println!(r#"cargo:rustc-cfg=k8s_pod_spec_supports_host_users"#);
182//!        }
183//!
184//!        // or emit new source code files under OUT_DIR, or anything else a build script can do.
185//!    }
186//!    ```
187//!
188//!    With these cfgs, the two cases above would be solved like this:
189//!
190//!    - ```rust,ignore
191//!      // The compile_error!() is only emitted if 1.24 or lower is selected.
192//!      #[cfg(not(k8s_pod_spec_supports_host_users))]
193//!      compile_error!("This crate requires the v1_25 (or higher) feature to be enabled on the k8s-openapi crate.");
194//!
195//!      ...
196//!
197//!      let pod_spec = k8s_openapi::api::core::v1::PodSpec {
198//!          host_users: ...,
199//!          ...
200//!      };
201//!      ```
202//!
203//!    - ```rust,ignore
204//!      let pod_spec = k8s_openapi::api::core::v1::PodSpec {
205//!          #[cfg(not(k8s_pod_spec_supports_host_users))]
206//!          host_users: ...,
207//!          ...
208//!      };
209//!      ```
210//!
211//! Note that both approaches require your crate to have a direct dependency on the `k8s-openapi` crate. Neither approach is available if your crate
212//! only has a transitive dependency on the `k8s-openapi` crate.
213//!
214//! The macros approach is easier to use since it doesn't require a build script.
215//!
216//! The build script method lets you emit arbitrary cfgs, emit arbitrary source code, and generally gives you more options, at the cost of needing a build script.
217//! `cfg()`s can be used in places where macros cannot, such as how the second example above shows it being used on a single field in a struct literal.
218//!
